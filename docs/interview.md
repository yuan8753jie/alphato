# AlphaTo 产品访谈记录

## 2026-03-24 第一次访谈

### 创始人原始构想

**用户：** 想要运营自己/他人社交媒体账号的人——可能是企业的营销专员，可能是一个Agency的社交媒体运营，也可能是个自媒体博主，甚至可以是一个AI。

**核心能力：** 帮助用户定义选题，追踪热点，预测行业/社会的大事件，然后把选题写成脚本，按照不同平台的风格，生成图文+视频。

**价值主张：** 未来一个人得到一个idea，不应该单单是一个文字版的Brief，应该是想法+选题+基本可看的Demo。初级用户觉得不错就拿来发作品了；高级用户可以把这些Demo发给自己的供应商，再refine出更棒的作品。

**三端形态：**
- **Agent端：** 找选题、生图、生视频的工具做成MCP/Skill，供其他AI调用
- **手机端：** 碎片时间查看信息，通过简单操作判定AI提供的idea/demo是好/不好/待定
- **桌面Web端：** 规划型用户，看账号近1~4周应该排布怎样的选题

### 第一轮访谈：用户与场景

**PM提问：**

Q1 — 你自己的经历：你本人是否有运营社交媒体账号的经历？实际操作中最痛的点是什么？是"不知道发什么"、"知道发什么但生产太慢"、还是"产出了但效果无法预判"？

Q2 — 第一个用户画像：如果产品明天就能用，你脑中第一个会拿来用的人是谁？（具体到：运营几个账号、什么平台、什么品类、团队几个人、现在用什么工具链）

Q3 — "Demo"的定义：一个美妆品牌想在小红书发一篇种草帖，AlphaTo产出的Demo具体长什么样？是一张配好文案的图？一个15秒的粗剪视频？还是一个包含多个方案的storyboard？

Q4 — 平台优先级：V1要支持哪些平台？有没有明确的优先级排序？

**创始人回答：**

A1 — 最痛的点是**不知道发什么**，尤其是运营企业账号时，还要跟企业各种核对，有很多限制。另一个痛点是**想法转变为Demo的人力成本高**，导致没法比较具象地看到更多可能性。

A2 — 第一个用户就是自己。有一个10人团队，为不同企业运营**数千个账号**。自己也有账号。平台分布：抖音社媒最多、抖音千川其次、然后是小红书。每天想idea抓热点非常花时间。
- **现有工具链：**
  - 抖音矩阵宝（账号管理）
  - 各平台搜罗热门话题/热点/大众新闻/行业新闻
  - Clipo（clipo.cc）— 自研混剪工具，用于批量生产数百条类似又能过审的视频
  - 可灵/即梦 + 素材 + 剪映 — 更高级需求的精细化处理
- **核心目标：** 在平台规则限制内，让账号内容更有流量

A3 — Demo的定义：
- **图文类：** 已经做好的图片+文字，相信当前生成技术已经够好
- **视频类：** 过程中应有：① 脚本表 → ② 分镜+脚本表 → ③ 粗剪样片。素材库有合适素材就用素材，没有就AI生成，甚至全AI生成也可以
- 创始人手上有视频样片想分享

A4 — 平台优先级：抖音 > TikTok > 小红书 > Instagram > 快手 > 微信视频号 > YouTube > Bilibili

### 第三轮访谈：V1 核心工作流

**PM提问：**

Q9 — 模拟了一个使用场景（运营小张周一早上打开Web看推荐选题→挑选→生成Demo→发客户），问场景是否正确，以及：系统主动推荐 vs 用户搜索？一次推荐多少？选题到Demo等多久？

Q10 — 账号"定位"信息从哪来？用户手动填？系统自动分析？

Q11 — 你团队现在"找选题"的真实流程是什么样的？

**创始人回答：**

A9 — 场景基本正确，但设想更大胆：

**Proactive 自主模式：**
- 系统不只推荐选题，还**自动评估**：为每个账号/产品/品牌定义多个目标受众角色（Persona）
- 这些虚拟受众对选题进行 Review
- Review 结果差 → 自动放弃
- Review 结果好 → 自动写脚本 → 多角色 Review 脚本 → 结果好的直接文生图/视频 → 产出粗剪 Demo
- 整个流程是**自治的**，用户打开时看到的是已经做好的 Demo

**数量与时间：**
- 每天 5~10 个选题推荐，未来可让用户自定义（Proactive 模式消耗 Token 不小）
- **Reactive 模式**（用户主动操作/对话）：选题→Demo **5分钟内**
- **Proactive 模式**（提前自动生成）：可以放宽到 **20~30 分钟**，好作品值得等

A10 — 账号定位有三种输入方式：
1. **手动填写**：告诉 AI 这是个怎样的账号
2. **绑定自己的账号**：系统自动分析
3. **给出对标账号**：系统学习对标

A11 — 现有选题工作流（极其重要的业务洞察）：

**信息源（散布在多处）：**
- 拼命刷各平台
- 抖音热点宝 / TikTok 热点
- 跨平台搜索（其他社交平台）
- 研究精品对标账号
- 历史上的今天
- 品类冷知识
- 最新的表现形式/创意趋势
- 自己脑暴
- 新闻 / 行业大事件
- 关注的热点账号动态

**团队协作流程：**
1. 团队成员**各自刷信息源**，一边看一边关联自己负责的客户账号/产品/活动是否合适
2. 发现合适的就**丢到聊天群**里，大家实时可见
3. 每天早上有**脑暴会**，会前值班同学把群里内容整理到**"选题表"**
4. 脑暴会上一起看其他人发掘的热点，判断：
   - 哪些热点适合**大多数客户**去蹭
   - 哪些热点**特别适合某个客户**
5. 修改选题表的状态
6. 对齐好的选题填入各客户的**营销日历**（也是个表）

### 第四轮访谈：深挖核心机制

**PM提问：**

Q12 — 虚拟受众 Review 机制：为一个账号定义几个受众角色？谁来定义？Review 判断标准是什么？

Q13 — 选题表和营销日历现在用什么工具管理？AlphaTo 是替代还是集成？

Q14 — 一个账号同时有多少选题在流水线上？一天发几条？选题的生命周期是什么？

**创始人回答：**

A12 — 不会自己定义受众角色，**希望 AI Agent 自动定义**。认为大语言模型足够聪明，能够定义出合理的样本空间来做受众 Review。创始人自认对市场研究不熟悉，希望把这个能力交给 AI。

A13 — 目前都用**飞书**。理解上应该是**替代**这些工具——团队太多时间花在做表做周报上了。非常希望产品能让伙伴直接**分享页面给客户或协作伙伴**，而不是只靠飞书。但也认同初期**可以先靠集成快速跑通**。

A14 — 一个账号一般**一天发1条**，偶尔2条。但也在思考：如果有个工具能让一个账号**一天发100条**不同的、都很有意思、对受众有益的内容，那也不错。

**PM追问（内嵌回答）：**

关于 Q12 — 虚拟受众自动生成：完全同意。LLM 做 Persona 定义+模拟 Review 是成熟的能力。系统可以基于账号定位自动生成 3~5 个典型受众 Persona，每个 Persona 按"会不会看完/会不会互动/会不会转化"打分。用户也可以手动调整 Persona 库。

关于 Q13 — 集成 vs 替代：建议 V1 走**飞书集成 + 自有轻量日历视图**双轨。理由：(1) 你团队已经在飞书上有成熟的协作习惯，强行替代会增加迁移成本；(2) 但"分享页面给客户"这个需求说明外部协作者不一定在飞书，自有 Web 页面从 V1 就该有，哪怕极简。

关于 Q14 — 日发100条：这本质上是把 AlphaTo 从"辅助创作工具"推向"内容工厂"。这和你之前说的 Clipo 矩阵裂变思路其实是一脉相承的——只不过 Clipo 是同一个视频裂变100条，而你现在设想的是100条**不同的**原创内容。这是一个非常好的远期愿景，但 V1 建议先跑通"日产5~10选题 → 人类筛选 → 日发1~2条精品"的流程。

### 第五轮访谈：产品边界与商业模式

**PM提问：**

Q15 — V1 的终点：到"产出Demo给人看"就结束，还是要包含"一键发布到抖音"？

Q16 — 多客户管理：不同客户的账号在系统里是隔离空间还是统一面板？

Q17 — 谁为这个产品付钱？付费模式怎么想的？

**创始人回答：**

A15 — V1 做到**"产出 Demo 给人看"就结束**，不需要一键发布到抖音。

A16 — 账号的规划、内容应该是**两个空间**；而新推荐的选题可以**共用**（或者直接一个选题给到多个账号）。希望听 PM 的建议。

A17 — 两个付费大头：
1. **Prosumer**（包含自己运营团队的同学）
2. **企业**（产品跑通后卖给企业）

定价思路：按消耗的 Token + 1~2 倍毛利卖给用户。自认不擅长定价，希望听专业意见。

**PM建议（内嵌回答）：**

**关于 Q16 — 多账号与选题的关系：**

建议三层架构：

```
第一层：热点池（Global Topic Feed）
  → 全局共享，AI 自动抓取的所有热点/趋势
  → 所有用户/账号都能看到

第二层：账号工作区（Account Workspace）
  → 每个账号有自己的定位、受众 Persona、营销日历
  → 系统自动计算"这个热点和这个账号的匹配度"
  → 高匹配的自动进入该账号的选题候选

第三层：内容流水线（Content Pipeline）
  → 每个账号独立的：选题 → 脚本 → Demo
```

这样的好处是：
- 一个热点可以同时匹配给多个账号，但每个账号的脚本/Demo 是基于自己的调性独立生成的
- 这完美映射了你团队现在的工作流——晨会上就是在做"哪些热点适合大多数客户、哪些特别适合某个客户"
- 操作上，运营可以手动"把一个选题推送给某几个账号"，也可以让系统自动推荐

**关于 Q17 — 定价建议：**

纯 Token 加价模式有几个问题：
1. **用户成本不可预测** — Proactive 模式下 Token 消耗波动大，用户每月账单不稳定，会产生焦虑
2. **抑制使用** — 用户会因为怕花钱而不敢开 Proactive，但 Proactive 恰恰是你产品最强的差异化
3. **企业采购不友好** — 企业喜欢固定预算

建议 **阶梯订阅 + 用量包** 混合模式：

| 层级 | 月费 | 包含内容 | 超出部分 |
|------|------|---------|---------|
| Starter | ¥99~199 | 3个账号，每日5选题推荐，Reactive模式 | 按量付费 |
| Pro | ¥499~799 | 10个账号，Proactive模式，每日10选题+自动Demo | 按量付费（折扣价） |
| Enterprise | 按需报价 | 无限账号，API/MCP接入，私有化部署选项 | 含量谈判 |

核心思路：
- 用**订阅锁定关系**，用**用量包控制毛利**
- Proactive 模式作为 Pro 档的核心卖点，倒逼用户升级
- V1 阶段不用纠结定价，先让自己团队用起来，等有了真实的 Token 消耗数据再定

### 第六轮访谈：技术与实现约束

**PM提问：**

Q18 — 技术栈偏好？

Q19 — AI 模型选择？

Q20 — V1 的时间预期？

**创始人回答：**

A18 — 一般用 **Next.js + Vercel + Supabase**。希望先听技术建议。

A19 — AI 模型思路：
- **抖音脚本/拆解：豆包大模型**（最懂短视频和国内的梗）
- **海外文案：** 不熟悉，想听推荐（提到 Grok？）
- **图片生成：Nano Banana**
- **视频生成：Kling（可灵）**

A20 — 预期**1~2周**出 V1。参考：创始人之前做了 makaron.app，过年4~5天就做出来了。全力和 AI 一起编程。

**PM建议（内嵌回答）：**

**关于 Q18 — 技术栈：**

Next.js + Vercel + Supabase 非常适合这个项目，同意。补充建议：

| 层 | 选择 | 理由 |
|---|------|------|
| 前端 | Next.js (App Router) | 你熟悉，SSR+RSC 适合 SEO 和分享页 |
| 部署 | Vercel | 快速部署，Edge Functions 覆盖全球（TikTok/Instagram 用户） |
| 数据库 | Supabase (PostgreSQL) | 实时订阅（选题状态变更通知）、Row Level Security（多客户隔离）、自带 Auth |
| 后台任务 | Vercel Cron + Supabase Edge Functions 或 Inngest | Proactive 模式需要定时跑 AI 流水线，不能靠前端触发 |
| 文件存储 | Supabase Storage 或 Cloudflare R2 | 存生成的图片/视频 Demo |
| MCP/Agent 层 | 独立的 API Routes + MCP SDK | 为 Agent 端暴露标准接口 |

关键点：**Proactive 模式是个后台 pipeline**，每天凌晨或定时跑"抓热点→匹配账号→虚拟受众评估→生成脚本→生成Demo"。Vercel 的 serverless 有执行时长限制（Hobby 10s，Pro 300s），如果 Pipeline 流程长，可能需要用 Inngest 或 Supabase Edge Functions 做长时任务编排。

**关于 Q19 — AI 模型：**

同意豆包做国内短视频脚本，它确实最懂国内语境。

海外文案推荐：
- **Claude** — 创意写作最强，语感自然，适合品牌调性要求高的内容
- **GPT-4o** — 全面均衡，多语言能力好
- **Grok** — 实时性强（接入 X/Twitter 数据），适合蹭海外热点，但纯文案质量不如前两者

建议：**海外选题/热点用 Grok 的实时能力，文案生成用 Claude**。可以做成可配置的，用户选模型。

图片/视频生成（Nano Banana + Kling）：OK，这些都通过 API 调用，架构上做成可插拔的 Provider 就行。

**关于 Q20 — 1~2 周：**

以你做 makaron.app 的速度，1~2 周出一个能用的 V1 是可行的，但需要非常克制地砍 scope。我的建议是把 V1 再拆成 V0.1 和 V0.5：

- **V0.1（第1周）：** 核心跑通——配置1个账号 → 手动触发选题生成 → 看到选题列表 → 选中1个 → 生成脚本 → 生成图文Demo。纯 Reactive 模式，Web 端，不做手机适配。
- **V0.5（第2周）：** 加入 Proactive 模式 + 虚拟受众评估 + 多账号 + 营销日历视图 + 手机端基础适配。

### 第八轮：补充与修正

**创始人补充：**

**账号配置不只是人设/调性，还需要存储丰富的上下文：**
1. **品牌调性与规则** — 品牌的风格、红线、规范
2. **推广产品列表** — 这个账号会推哪些产品
3. **产品资料** — 详细的产品信息（出名产品可用搜索补全）
4. **产品三视图/白底图** — 供生成图/视频时将产品融合进画面

**关于 Proactive Pipeline：**
创始人认为 Proactive 是最核心的思想——"晚上已经有人帮忙把活干了一大半"是非常奇妙的体验。愿意挑战 Vercel 的长任务限制，希望 PM/技术指导实现。

**PM回应：**

关于账号配置——这非常关键。账号不是一个简单的"标签"，而是一个**知识库**。重新定义账号工作区的数据模型：

```
Account Workspace = {
  基础信息: { 名称, 平台, 账号链接 },
  品牌上下文: { 品牌名, 调性描述, 品牌规则/红线 },
  产品库: [
    {
      产品名,
      产品描述,
      卖点,
      产品图片(三视图/白底图),
      相关链接
    }
  ],
  受众Persona: [ AI自动生成 + 用户可调 ],
  对标账号: [ 账号链接列表 ],
  营销日历: { 排期表 }
}
```

这意味着生成脚本/Demo 时，AI 不只是"写一段抖音文案"，而是在完整的品牌+产品上下文中创作，甚至可以把产品白底图合成到生成的视频/图片里。这是一个非常强的差异化。

关于 Proactive Pipeline 的 Vercel 实现策略——完全可以做。核心思路是**任务拆分+链式调用**：

1. Vercel Cron 定时触发（比如每天凌晨2点）
2. 第一个 Serverless Function：抓热点，结果写入 Supabase，触发下一步
3. 第二个 Function：读取热点 × 账号，做匹配+虚拟受众评估，结果写入 DB
4. 第三个 Function：为通过评估的选题生成脚本
5. 第四个 Function：调用图片/视频 API 生成 Demo
6. 每一步都在 Supabase 记录状态，任何一步失败可以重试

每个 Function 控制在 Vercel Pro 的 300s 限制内，通过 DB 状态流转串联。用户早上打开看到的是已经流转完毕的结果。

### 第九轮：产品质量观

**创始人补充：**

AlphaTo 跟传统工具不一样。传统工具只要页面能点、表单能填、列表信息无误就行了。AlphaTo 的核心是**效果**——选题得好、脚本得好、图得好、视频得好。不需要完美，但 AI 至少得做到 **60~70 分**，不能错得离谱。**效果是这个产品的核心。**

**PM回应：**

这是一个非常关键的认知，它从根本上改变了开发策略和验收标准。

传统 SaaS 的验收是：功能能用 → 交互顺畅 → 上线。
AlphaTo 的验收必须是：功能能用 → 交互顺畅 → **产出质量达标** → 上线。

这意味着我们的开发工作有两条并行线：

**线1：工程线（传统开发）**
- 页面、交互、数据流、API 对接
- 这部分相对可控，1~2周能搞定

**线2：效果线（Prompt Engineering + 模型调优）**
- 选题推荐的准确度和创意水平
- 脚本的可用度（结构、语言、梗的运用）
- 图片的品质和品牌一致性
- 视频的观感（节奏、卡点、画面质量）
- 这部分需要**大量的试错和迭代**，是真正的核心壁垒

建议在 PRD 中定义每个环节的**质量基线**：

| 环节 | 60分（最低可接受） | 80分（良好） |
|------|-------------------|-------------|
| 选题 | 和账号品类相关，蹭到了真实热点，不离谱 | 角度新颖，有传播潜力，团队会说"这个不错" |
| 脚本 | 结构完整(开头hook+中间+结尾CTA)，语言通顺，时长合理 | 有记忆点，语言有抖音感，能直接拿去拍 |
| 图片 | 主题相关，画面清晰，文字可读 | 风格统一，有品牌感，产品融合自然 |
| 视频 | 画面连贯，节奏基本合理，不出戏 | 卡点准确，转场流畅，口播自然 |

开发策略调整：**V0.1 应该先只跑通选题→脚本这两个环节的效果线**，因为这是纯文本，迭代最快、成本最低。图片和视频的效果调优放到 V0.5，因为每次生成都要花钱花时间，不适合高频迭代。

验收方式建议：每个阶段拿 **你团队的真实账号** 跑3~5个 case，你和团队成员打分。如果平均分低于60，就调 prompt 继续迭代，不进入下一阶段。
